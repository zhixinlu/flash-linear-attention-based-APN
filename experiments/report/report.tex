\begin{abstract}
We benchmark three sequence model architectures on the sequential CIFAR-10 task:
the \textbf{Associative Plasticity Network (APN)}, mapped onto the gated delta-rule Triton kernel;
\textbf{DeltaNet}, a delta-rule linear attention model; and a standard \textbf{softmax Transformer} with FlashAttention-2.
All experiments use 10 layers, run for 200 epochs on a single NVIDIA H200 GPU, and share the same outer architecture
(input projection $\to$ residual layer stack $\to$ classifier).
We compare accuracy, training speed, GPU memory, and parameter efficiency across
width-matched, parameter-matched, and FFN-augmented configurations.
Key findings: (1) plain APN (103K params) achieves 58.3\% test accuracy---within 5 points of the
Transformer (63.5\%, 1.2M params)---while training 4$\times$ faster and using 2.7$\times$ less GPU memory;
(2) APN is uniquely resistant to overfitting, with near-zero peak-to-final accuracy drop;
(3) adding an FFN to APN \emph{hurts} performance (56.8\%) despite 12$\times$ more parameters,
while DeltaNet+FFN (62.8\%) nearly matches the Transformer.
These results suggest APN's constrained $D^2$-parameter design acts as an implicit regularizer
that does not benefit from naive capacity scaling.
\end{abstract}

\section{Introduction}

The Associative Plasticity Network (APN) is a biologically-inspired recurrent model that maintains a Hebbian plastic memory matrix $M_t$, updated at every time-step.
By mapping the APN recurrence onto the \emph{gated delta-rule} chunk kernel from the Flash Linear Attention (FLA) library, we achieve efficient GPU training with $O(TD)$ complexity instead of the $O(T^2 D)$ cost of standard softmax attention.

This report summarizes all experiments conducted on the \textbf{Seq-CIFAR-10} benchmark, where each $32 \times 32$ CIFAR-10 image is flattened into a sequence of $T = 1024$ RGB pixels ($3$-dimensional features), and the model classifies from the last time-step.
We also provide a detailed explanation of the GPU kernel infrastructure and chunk algorithm (Section~\ref{sec:kernels}) that makes efficient training possible.

\section{Model Architectures}

All three models share the same outer wrapper:
\[
\texttt{Linear}(3 \to D) \;\to\; \tanh \;\to\; \underbrace{[\texttt{Layer}_i + \text{residual}]}_{i=1}^{L} \;\to\; \texttt{LayerNorm} \;\to\; \texttt{Linear}(D \to 10)
\]

\subsection{APN Layer}

Each APN layer maintains a plastic state matrix $S_t \in \mathbb{R}^{D \times D}$ with the gated delta-rule update:
\begin{equation}
S_t = \lambda \, S_{t-1} + \frac{\eta(1-\lambda)}{D} \, \mathbf{k}_t \otimes \bigl(\mathbf{v}_t - S_{t-1}^\top \mathbf{k}_t\bigr)
\end{equation}
where $\mathbf{k}_t = \text{act}(\mathbf{x}_t)$, $\mathbf{v}_t = W \cdot \text{act}(\mathbf{x}_t)$, $\lambda \in (0,1)$ is a trainable decay, and $\eta$ is a trainable learning rate scalar. The output is $\mathbf{o}_t = \mathbf{k}_t^\top S_t + \mathbf{v}_t$ (static path plus dynamic recall).

\textbf{Parameters per layer:} $D^2$ (one linear projection $W$) $+ 2D$ (LayerNorm) $+ 2$ (scalars $\eta, \lambda$) $\approx D^2$.

\subsection{DeltaNet Layer}

DeltaNet uses separate Q/K/V projections with the (non-gated) delta rule:
\begin{equation}
S_t = S_{t-1} + \beta_t \, \mathbf{k}_t \otimes \bigl(\mathbf{v}_t - S_{t-1}^\top \mathbf{k}_t\bigr), \quad \mathbf{o}_t = \mathbf{q}_t^\top S_t
\end{equation}
where $\beta_t = \sigma(W_\beta \mathbf{x}_t)$ is a learned, input-dependent step size.

\textbf{Parameters per layer:} $3D^2$ (Q/K/V projections) $+ D$ ($\beta$ projection) $+ 2D$ (LayerNorm) $\approx 3D^2$.

\subsection{Transformer Layer}

A pre-norm causal Transformer block:
\[
\texttt{LN} \to \texttt{MHA} \to \text{residual} \to \texttt{LN} \to \texttt{FFN}(D \to 4D \to D) \to \text{residual}
\]
Self-attention uses \texttt{F.scaled\_dot\_product\_attention} with \texttt{is\_causal=True}, which auto-dispatches to FlashAttention-2 on CUDA.

\textbf{Parameters per layer:} $3D^2$ (QKV) $+ D^2$ (output proj) $+ 8D^2$ (FFN) $+ 4D$ (two LayerNorms) $\approx 12D^2$.

\subsection{Optional FFN for APN/DeltaNet}

To isolate the effect of the attention mechanism from the FFN, we support an optional \texttt{--use-ffn} flag that adds a pre-norm FFN block (identical to the Transformer's) after each APN or DeltaNet layer. This adds $\approx 8D^2$ parameters per layer.

\textbf{Why APN/DeltaNet can be faster.}
For $T=1024$, $D=100$: the softmax attention FLOPs scale as $T^2 D = 1024^2 \times 100 \approx 10^8$, while the chunked delta-rule scales as $TCD = 1024 \times 64 \times 100 \approx 6.5 \times 10^6$ --- a $\approx 15\times$ reduction in the attention-like computation. However, the APN/DeltaNet kernels are Triton-based (less mature than FlashAttention's hand-tuned CUDA), and at small $D$ the linear projections dominate, so the wall-clock advantage depends on the specific configuration.

For \textbf{inference}, the advantage is even clearer: APN/DeltaNet maintain a fixed $D \times D$ state (constant memory), while the Transformer's KV cache grows linearly with sequence length.
Section~\ref{sec:kernels} explains in detail how the FLA Triton kernels and chunk algorithm achieve these theoretical advantages in practice.

\section{Experimental Setup}

\begin{itemize}
\item \textbf{Hardware:} Single NVIDIA H200 GPU (141 GB HBM3e, 228 KB shared memory per SM)
\item \textbf{Task:} Seq-CIFAR-10 (1024-step sequences, 10-class classification)
\item \textbf{Optimizer:} AdamW ($\text{lr} = 10^{-3}$, weight decay $= 10^{-4}$, grad clip $= 1.0$)
\item \textbf{Schedule:} Cosine annealing (with optional linear warmup)
\item \textbf{Batch size:} 64
\item \textbf{Epochs:} 200 (unless noted otherwise)
\item \textbf{Seed:} 42
\end{itemize}

\subsection{Key Observations}

\begin{enumerate}
\item \textbf{APN with \texttt{act=none} is the best APN variant}, reaching 58.34\% (with warmup) versus 51--53\% for \texttt{act=tanh}. The identity activation (no nonlinearity on keys) allows the model to use the full input range without saturating gradients.

\item \textbf{APN is the most parameter-efficient model.} With only 103K parameters ($1\times$), APN (act=none) achieves 58.34\%. DeltaNet requires $3\times$ the parameters (305K) to reach 62.28\%, and the Transformer requires $12\times$ (1.2M) to reach 63.50\%. Per parameter, APN extracts \textbf{0.56\% accuracy per 1K params}, versus 0.20\% for DeltaNet and 0.053\% for the Transformer.

\item \textbf{APN is uniquely resistant to overfitting.} Table~\ref{tab:results} reveals a striking pattern in the ``Best vs.\ Final'' columns: DeltaNet $D=58$ peaks at epoch~18 then \emph{drops 4.7 points} by epoch 200 (60.06\% $\to$ 55.40\%); Transformer $D=29$ drops 3.9 points (59.75\% $\to$ 55.89\%); DeltaNet $D=100$ drops 1.5 points. In contrast, APN act=none peaks at epoch~189 with its final accuracy only \textbf{0.17 points} below peak. APN does not require early stopping---a significant practical advantage.

\item \textbf{Adding FFN hurts APN but helps DeltaNet.} This is the most surprising result. At $\approx$1.2M params:
\begin{itemize}
    \item \textbf{APN + FFN (56.84\%) is \emph{worse} than plain APN (58.34\%)} despite having $12\times$ more parameters. The FFN enables APN to reach 100\% train accuracy (vs.\ 62.9\% without FFN), but this capacity is entirely wasted on memorization---test accuracy \emph{decreases} by 1.5 points.
    \item \textbf{DeltaNet + FFN (62.80\%) nearly matches the Transformer (63.50\%)}---only 0.7 points behind at equal parameters. Moreover, DeltaNet+FFN stabilizes training: its peak occurs at epoch~198 (vs.\ epoch~11 without FFN) with \emph{zero} peak-to-final drop.
\end{itemize}
This suggests APN's single-projection design acts as an \textbf{implicit regularizer}: the $D^2$-parameter bottleneck constrains capacity in a way that prevents overfitting, but also prevents the model from utilizing additional FFN capacity. DeltaNet's richer $3D^2$ attention mechanism can leverage the FFN to close the gap with the Transformer.

\item \textbf{Warmup helps APN but hurts DeltaNet.} DeltaNet $D=100$ without warmup: 62.28\% peak; with 20-epoch warmup: 59.26\% peak---a 3-point \emph{drop}. For APN, warmup provides a modest but consistent gain (58.12\% $\to$ 58.34\%). Interestingly, both DeltaNet+FFN and APN+FFN \emph{do} benefit from warmup (both use 20-epoch warmup and reach stable plateaus).

\item \textbf{APN is the fastest per epoch.} At $\approx$26s/epoch, APN (act=none, $D=100$) is $\approx$4$\times$ faster than DeltaNet ($\approx$100s) and $\approx$1.5$\times$ faster than the Transformer ($\approx$39s) at the same width. Even APN+FFN (55.3s) is faster than DeltaNet without FFN.

\item \textbf{APN uses the least GPU memory.} 1,590 MB vs 2,133 MB (DeltaNet) vs 4,278 MB (Transformer) at $D=100$. With FFN, APN+FFN (4,358 MB) is comparable to the Transformer (4,278 MB)---the FFN dominates memory at this scale.
\end{enumerate}

\section{GPU Kernel Implementation and the Chunk Method}
\label{sec:kernels}

A critical contributor to APN/DeltaNet's practical speed is the \textbf{fused Triton kernel} implementation from the Flash Linear Attention (FLA) library.
This section explains the GPU execution model, contrasts a naive PyTorch implementation with the kernel-based approach, and details the chunk algorithm that enables efficient training.

\subsection{GPU Memory Hierarchy}

Modern GPUs have a layered memory system, where each level trades capacity for speed:

\begin{table}[h]
\centering
\caption{GPU memory hierarchy (H200).}
\label{tab:memhierarchy}
\begin{tabular}{lccc}
\toprule
\textbf{Level} & \textbf{Size} & \textbf{Bandwidth} & \textbf{Control} \\
\midrule
Registers & $\approx$256 KB / SM & Instantaneous & Per-thread \\
Shared Memory (SRAM) & 228 KB / SM & $\approx$20 TB/s & Programmer-managed \\
L2 Cache & 40--60 MB & $\approx$6 TB/s & Automatic \\
HBM (Global Memory) & 141 GB & $\approx$3 TB/s & Default storage \\
\bottomrule
\end{tabular}
\end{table}

Most GPU workloads are \textbf{memory-bound}: the arithmetic units can perform trillions of operations per second, but spend most time waiting for data from HBM.
The key to fast kernels is keeping data in shared memory / registers as long as possible to minimize HBM round-trips.

\subsection{Naive APN vs.\ Fused Kernel APN}

\textbf{Naive PyTorch implementation.} A straightforward APN loop issues 7+ kernel launches per timestep (activation, two matmuls, scale, outer product, add, copy):
\begin{itemize}
\item $T \times 7 = 7{,}168$ kernel launches per layer, each with $\sim$10\,$\mu$s overhead $\approx$ 72\,ms in launch overhead alone.
\item The $D \times D$ state matrix $S$ is read from / written to HBM at every timestep.
\item A Python for-loop serializes all timesteps, preventing the GPU from planning work ahead.
\item Backpropagation requires saving all $T$ intermediate states. For $B=64$, $D=100$, $T=1024$, $L=10$:
\[
B \times T \times D^2 \times 4\,\text{bytes} \times L = 64 \times 1024 \times 10^4 \times 4 \times 10 \approx 26\,\text{GB}.
\]
\end{itemize}

\textbf{Fused kernel implementation.}
The FLA library replaces this loop with a compact Triton kernel pipeline.
In our experiments, DeltaNet uses the \texttt{fused\_recurrent\_delta\_rule} kernel (a single-kernel scan), while APN uses the \texttt{chunk\_gated\_delta\_rule} operator (a multi-kernel pipeline computing gates, intra-chunk WY terms, inter-chunk state propagation, and outputs).
The improvements are:
\begin{enumerate}
\item \textbf{A handful of kernel launches} instead of 7{,}168, eliminating launch overhead.
\item \textbf{State tiles stay on-chip}: each Triton program owns one batch/head/tile of the $D \times D$ state in registers or SRAM. Between timesteps, the state is updated in-place without an HBM round-trip.
\item \textbf{No Python loop}: the entire forward pass executes autonomously on the GPU.
\item \textbf{Smart backward pass}: the chunk method avoids storing all $T$ intermediate states (see below).
\end{enumerate}
This is why the kernel-based APN is $10$--$100\times$ faster than the naive version and uses $\sim$1.6\,GB instead of $\sim$26\,GB.

\subsection{The Chunk Algorithm}
\label{sec:chunk}

The delta-rule recurrence $S_t = f(S_{t-1}, \mathbf{k}_t, \mathbf{v}_t)$ is inherently sequential: $S_t$ depends on $S_{t-1}$.
A fused recurrent kernel loops through all $T$ timesteps one by one; parallelism is limited to batch and state dimensions.

The \textbf{chunk method} splits the sequence into $T/C$ chunks of $C$ timesteps (default $C=64$) and restructures the computation:

\begin{enumerate}
\item \textbf{Intra-chunk (parallel across chunks).}
Within each chunk, the recurrence is reformulated as a lower-triangular linear system using the \textbf{WY representation}:
\[
L \, \tilde{\mathbf{v}} = \mathbf{v} - S_{\text{prev}} K^\top, \quad
L_{ij} = \begin{cases} 1 & i = j \\ -\beta_i \, \mathbf{k}_i^\top \mathbf{k}_j \prod_{m=j+1}^{i-1} g_m & i > j \\ 0 & i < j \end{cases}
\]
This triangular solve maps to GPU matrix operations and can be solved for all $T/C$ chunks simultaneously.

\item \textbf{Inter-chunk (sequential across chunk boundaries).}
The $D \times D$ state is propagated from chunk 0 $\to$ chunk 1 $\to \cdots \to$ chunk $T/C-1$.
For $T=1024, C=64$, this is only 16 sequential state updates instead of 1024.
\end{enumerate}

\begin{table}[h]
\centering
\caption{Comparison of execution strategies for $T=1024$, $C=64$.}
\label{tab:strategies}
\begin{tabular}{lccc}
\toprule
\textbf{Method} & \textbf{Sequential depth} & \textbf{HBM reads of $S$} & \textbf{Backward state storage} \\
\midrule
Naive Python loop & $T = 1024$ & $T \times D^2$ & All $T$ states \\
Fused recurrent & $T = 1024$ & 1 (load once) & Recompute \\
Chunk ($C=64$) & $T/C = 16$ & $T/C \times D^2$ & $T/C$ boundary states \\
\bottomrule
\end{tabular}
\end{table}

The chunk method's advantage grows with sequence length: at $T=8192$, the sequential depth is 128 (vs.\ 8192); at $T=65536$ it is 1024 (vs.\ 65536).
For the backward pass, only $T/C$ boundary states need to be stored---a $64\times$ reduction compared to the naive approach.

\subsection{Tile Alignment and Performance Cliffs}

The chunk kernel processes the $D \times D$ state in tiles (typically $64 \times 64$ blocks).
When $D$ is not a multiple of the tile size, the kernel must pad to the next tile boundary, wasting compute:

\begin{table}[h]
\centering
\caption{Effect of $D$ on tile count and efficiency (tile size $=64$).}
\label{tab:tiling}
\begin{tabular}{ccccc}
\toprule
$D$ & Tiles/dim & Tile blocks & Padding waste & Relative efficiency \\
\midrule
64 & 1 & 1 & 0\% & Best \\
100 & 2 & 4 & 22\% & Good \\
128 & 2 & 4 & 0\% & Best \\
173 & 3 & 9 & 10\% & Poor (tile jump) \\
256 & 4 & 16 & 0\% & Good \\
\bottomrule
\end{tabular}
\end{table}

For $D=100$: $\lceil 100/64 \rceil = 2$ tiles per dimension $\to 4$ tile blocks.
For $D=173$: $\lceil 173/64 \rceil = 3$ tiles $\to 9$ blocks, plus padding waste and reduced occupancy.
This explains the $\approx$10$\times$ slowdown from $D=100$ (26\,s/epoch) to $D=173$ (453\,s/epoch) observed in the APN tanh experiment---far worse than the $1.7^2 \approx 3\times$ ratio of $D^2$ arithmetic alone.

\textbf{Recommendation:} For efficient Triton kernel execution, prefer tile-aligned dimensions $D \in \{64, 128, 192, 256\}$ or small dimensions with only 2 tiles per axis ($D \le 128$).

\subsection{Why APN Is Faster Than DeltaNet and the Transformer}

Beyond the shared kernel infrastructure, three factors explain APN's additional speed advantage:

\begin{enumerate}
\item \textbf{Fewer projections.}
APN uses a single $D \times D$ projection per layer; DeltaNet uses three ($W_Q, W_K, W_V$); the Transformer uses four projections plus a $D \to 4D \to D$ FFN.
At $D=100$: APN computes 10K multiply-adds per layer; DeltaNet 30K; the Transformer 120K.

\item \textbf{Compact recurrent state.}
APN/DeltaNet store a $D \times D$ state ($\approx$40\,KB at $D=100$ in FP32).
The Transformer must store Q/K/V activations proportional to $T$ for the backward pass, even with FlashAttention-2's memory-efficient recomputation, resulting in higher HBM traffic.

\item \textbf{Linear vs.\ quadratic complexity in $T$.}
Softmax attention (even with FlashAttention) is $O(T^2 D)$; the chunked delta rule is $O(TCD + TD^2/C)$---linear in $T$.
At $T=1024, D=100$: attention FLOPs are $\approx 15\times$ fewer for APN.
\end{enumerate}

\textbf{Caveat:} FlashAttention-2 is implemented in hand-tuned CUDA, achieving near-peak throughput.
The FLA kernels are Triton-based (correct but not fully optimized for Hopper).
As $T$ grows beyond 4096, the $O(T)$ vs.\ $O(T^2)$ gap dominates regardless of kernel maturity.

\textbf{Note on Triton version:} We pin \texttt{triton>=3.3.0,<3.4.0} due to a known correctness bug in the gated chunk backward kernel on Hopper GPUs with Triton $\ge$3.4 (FLA issue \#640).
Triton 3.3 generates correct code and runs \emph{faster} on H200 (26\,s/epoch) than on A100 with Triton 3.6 (33\,s/epoch), as Hopper's raw hardware advantages (more SMs, higher bandwidth, 228\,KB shared memory) compensate for missing compiler optimizations.

\textbf{Why does the FFN hurt APN?} All three models with $\approx$1.2M parameters reach 100\% training accuracy, confirming they have sufficient capacity to memorize the dataset. The critical difference is in test generalization:
\begin{itemize}
\item The \textbf{Transformer} and \textbf{DeltaNet+FFN} generalize well despite memorizing: their attention mechanisms (softmax QKV or delta-rule QKV with separate Q/K/V projections) learn useful representations that the FFN can refine.
\item \textbf{APN+FFN} fails to generalize: the APN layer's single-projection design ($\mathbf{k} = \mathbf{v} = W \cdot \mathbf{x}$, sharing keys and values) produces representations that are too constrained for the FFN to improve. The FFN instead learns to memorize training-specific patterns.

\end{itemize}

This result highlights a fundamental architectural difference: DeltaNet's separate Q, K, V projections ($3D^2$) create a richer feature space that benefits from FFN post-processing, while APN's tied key-value design ($D^2$) does not.

\section{Ongoing Experiments: Gated Forgetting Recurrence (GFR)}
\label{sec:gfr}

We introduce a simplified recurrent architecture---the \textbf{Gated Forgetting Recurrence (GFR)}---that replaces APN's $D \times D$ matrix state with $K$ independent scalar memory slots per neuron, using the HGRN chunk kernel for an element-wise gated scan.

\subsection{GFR Layer Architecture}

Each GFR layer maintains $K$ memory slots per neuron (total state: a vector of $D \cdot K$ scalars):
\begin{equation}
m_{i,k,t} = \lambda_k \, m_{i,k,t-1} + z_{i,t}, \quad
h_{i,t} = z_{i,t} + \sum_{k=1}^{K} \eta_k \, m_{i,k,t}
\end{equation}
where $\mathbf{z}_t = W \cdot \text{act}(\mathbf{x}_t)$ is the projected input, $\lambda_k \in (0,1)$ is a per-slot decay rate, and $\eta_k$ is a per-slot readout weight.
The output passes through LayerNorm before the residual connection.

\textbf{Key differences from APN:}
\begin{itemize}
\item \textbf{State:} $D \cdot K$ scalars (vector) vs.\ $D \times D$ matrix. At $D=100, K=4$: 400 vs.\ 10,000 elements.
\item \textbf{Recurrence:} Element-wise multiply-add (trivially parallel) vs.\ rank-1 outer product + delta correction.
\item \textbf{Kernel:} \texttt{chunk\_hgrn} (simple prefix scan) vs.\ \texttt{chunk\_gated\_delta\_rule} (WY transform + triangular solve).
\item \textbf{No tiling sensitivity:} HGRN operates on flat vectors, not $D \times D$ tile blocks---no performance cliffs at non-aligned $D$.
\end{itemize}

\textbf{Parameters per layer:} $D^2$ (projection $W$) $+ 2K$ (scalars $\lambda, \eta$) $+ 2D$ (LayerNorm) $\approx D^2$.

\subsection{Parameter Granularity Options}

The GFR implementation supports multiple granularities for $\lambda$ and $\eta$:

\begin{table}[h]
\centering
\caption{GFR parameter granularity options. $D=100$, $K=4$.}
\label{tab:gfr_granularity}
\begin{tabular}{llccc}
\toprule
\textbf{Mode} & \textbf{CLI Flags} & $\boldsymbol{\lambda}$ \textbf{shape} & $\boldsymbol{\eta}$ \textbf{shape} & \textbf{Extra params/layer} \\
\midrule
Per-slot (default) & (none) & $[K]$ & $[K]$ & 8 \\
Per-neuron-per-slot & \texttt{--gfr-per-neuron-lam/eta} & $[D, K]$ & $[D, K]$ & 800 \\
\bottomrule
\end{tabular}
\end{table}

In per-slot mode, all $D$ neurons within a slot share the same $\lambda_k$ and $\eta_k$.
In per-neuron mode, each of the $D \cdot K = 400$ scalar memories has its own learned decay and readout weight, initialized from the per-slot defaults but free to diverge during training.
The HGRN chunk kernel is agnostic to this distinction---it processes a flat $D \cdot K$ vector regardless.

\subsection{Speed Comparison with APN}

GFR should be $\approx$1.5--1.7$\times$ faster than APN per epoch at the same $D$:
\begin{itemize}
\item Both share the same $D \times D$ linear projection (dominates wall-clock time at $D=100$).
\item APN's recurrence kernel (\texttt{chunk\_gated\_delta\_rule}) processes a $D \times D$ state matrix with tiled outer products and WY triangular solves; GFR's \texttt{chunk\_hgrn} kernel does a simple element-wise gated scan on $D \cdot K = 400$ elements---effectively free ($< 0.1$\,ms at $T=1024$).
\item GFR avoids the Triton $\ge$3.4 backward correctness bug (no gated chunk delta rule kernel involved).
\end{itemize}

\section{Future Work}

Based on these findings, several directions could improve APN's performance:

\begin{enumerate}
\item \textbf{Separate key and value projections for APN.} The current APN design ties $\mathbf{k}$ and $\mathbf{v}$ through a single projection. Decoupling them (e.g., $\mathbf{k} = W_K \mathbf{x}$, $\mathbf{v} = W_V \mathbf{x}$) would increase per-layer cost to $2D^2$ but may enable the model to benefit from FFN augmentation.
\item \textbf{Regularization for APN+FFN.} Dropout, weight decay, or stochastic depth may prevent the FFN from memorizing when paired with APN.
\item \textbf{Multi-head APN.} Splitting $D$ into $H$ heads with $D/H$-dimensional states may improve representation diversity.
\item \textbf{Activation sweep.} Testing \texttt{sigmoid}, \texttt{softsign}, \texttt{silu}, and \texttt{gelu} activations on APN keys.
\item \textbf{Longer sequences.} Scaling to $T > 4096$ where the $O(T)$ vs.\ $O(T^2)$ gap dominates and APN/DeltaNet's linear complexity advantage becomes decisive.
\item \textbf{Language modeling.} Evaluating APN on autoregressive language tasks where the Hebbian memory matrix may better capture associative patterns.
\end{enumerate}

\section{Conclusions}

\begin{enumerate}
\item \textbf{Plain APN is the most parameter-efficient model.} With only 103K parameters, APN (act=none) achieves 58.34\% test accuracy---within 4 points of DeltaNet (62.28\%, 305K params) and 5 points of the Transformer (63.50\%, 1.2M params). Per 1K parameters, APN delivers $3\times$ the accuracy gain of DeltaNet and $11\times$ that of the Transformer.

\item \textbf{APN is uniquely resistant to overfitting.} APN's peak-to-final accuracy drop is only 0.17 points, compared to 4.7 points for DeltaNet $D=58$ and 3.9 points for Transformer $D=29$ at the same parameter budget. APN does not require early stopping.

\item \textbf{Adding FFN does not help APN.} APN+FFN (56.84\%, 1.2M params) is \emph{worse} than plain APN (58.34\%, 103K params) by 1.5 points. The FFN enables 100\% training accuracy but this capacity is entirely wasted on memorization. APN's single-projection design acts as an implicit regularizer that cannot leverage additional capacity through naive FFN augmentation.

\item \textbf{DeltaNet+FFN nearly matches the Transformer.} At $\approx$1.2M params, DeltaNet+FFN reaches 62.80\% vs.\ the Transformer's 63.50\%---only 0.7 points behind. Moreover, DeltaNet+FFN is the most stable model in the 1.2M group (0.00-point peak-to-final drop).

\item \textbf{APN trains $\approx$4$\times$ faster} than DeltaNet and $\approx$1.5$\times$ faster than the Transformer per epoch at $D=100$, using $\approx$2.7$\times$ less GPU memory.

\item \textbf{The identity activation (\texttt{act=none}) is best for APN}, outperforming \texttt{tanh} by 5--6 points. Warmup helps APN (+0.2 points) but hurts DeltaNet ($-$3 points).

\item \textbf{The key architectural bottleneck} for APN is its tied key-value representation ($\mathbf{k} = \mathbf{v} = W\mathbf{x}$). Decoupling keys and values---moving from $D^2$ to $2D^2$ parameters per layer---is the most promising direction to close the accuracy gap while preserving APN's speed and memory advantages.
\end{enumerate}
